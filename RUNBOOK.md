```bash
cd document-ai && npm run dev:server     # :8787 — holds MISTRAL_API_KEY
cd frontend     && npm run dev           # :5173 — Relire, "Document AI" tab
```

```bash
cd document-ai && MOCK_OCR=1 npm run dev
```

## You never pay for the same document twice

Results are cached on disk (`document-ai/.cache/`, gitignored), keyed by the **content of the
document** plus the settings that change the response.

- Press **Run** on a document you've already processed → served from disk in ~0.04s, **no API
  call, nothing billed**. The chip says `⚡ cached · free`.
- Survives a page reload, a server restart, and a different browser.
- Keyed by **content, not filename** — rename the file, or drop the same scan in twice, still a hit.
- **Chunks are cached individually**, so an interrupted 9-chunk run resumes for free: the chunks
  that already went through are hits; only the missing ones are paid for. Re-running after a
  failure halfway is cheap.
- Table mode and the Header/Footer chips are applied client-side, so changing them does **not**
  bust the cache. Changing the model, page range, bounding boxes, confidence, or the annotation
  schema does — because those change what the API returns.

**To deliberately pay for a fresh run:** the separate `↻ Force re-run` button. It's separate on
purpose — the button you hit by reflex must be the free one.

```bash
curl -s localhost:8787/api/cache     # {"entries":N,"bytes":N}
rm -rf document-ai/.cache            # start paying again
```

### The cache never hit for books, and the bill was silent (fixed July 2026)

**Symptom:** small PDFs showed `⚡ cached · free` on a re-run; a real book — anything over the
4 MB inline limit — **never did**, no matter how many times you uploaded it. `.cache/` filled up
with hundreds of entries that were never read back.

**Cause:** a book doesn't go to the server as your PDF. It goes as chunks *built* by
`web/src/lib/split.ts`, and pdf-lib's `PDFDocument.create()` stamps a fresh
`CreationDate`/`ModificationDate` into every document it makes. So the same page, rebuilt a
second later, serialised to **different bytes** — and the cache key is a hash of exactly those
bytes. Every chunk of every upload missed. The book you had already paid to OCR was re-billed on
every re-upload, quietly, and the only visible symptom was a chip that never appeared.

```
same page, built twice:  55424905ff6332c1  ≠  881fc5c0354ebfe4     # default
                         d562cba09d55dfe5  =  d562cba09d55dfe5     # updateMetadata: false
```

**Fix:** `PDFDocument.create({ updateMetadata: false })`. Determinism of the chunk build is a
**correctness property of the cache**, not a nicety — `split.test.ts` builds the same chunk twice
and asserts the bytes are identical, so it can't regress silently again.

**The lesson worth keeping:** a cache that misses doesn't fail loudly, it just costs money. If a
key is a hash of generated bytes, something has to pin those bytes.

## Symptoms

### "fetch failed" / the run dies on a big PDF

**Not your file, and not the app.** Mistral resets the connection mid-upload for large request
bodies. It is intermittent and worsens with size — measured minutes apart:

```
20MB -> EPIPE      13MB -> ECONNRESET      20MB -> HTTP 200
```

It reproduces **identically under `curl`**, so it is the network path, not Node, not undici,
not our code. Retries with backoff do not rescue it.

> Before blaming the client, reproduce with a different HTTP stack. One `curl` settled this.

**This is already handled:** PDFs over 4 MB are split client-side (`web/src/lib/split.ts`) and
only the pages being OCR'd are sent, ~2.5 MB per chunk. A "pages 1-4" run of a 266-page book
uploads four pages. **Do not "simplify" this back into one big upload.**

If it still fails, shrink the chunk: `TARGET_CHUNK_BYTES` in `split.ts`.

### The page indicator says "9 chunks"

That is **success**, not a warning. It's how the run worked. It appears as a chip (`⑂ 9 chunks`),
never in the orange banner — that banner is reserved for things you may need to act on.

### The app is dead / nothing loads

Check the ports first (above). The most likely cause is the most boring one: the server you're
looking at isn't running. `:5174` and `:5173` are **different apps**; killing one doesn't kill
the other, and a restart of the API server doesn't restart either front end.

### "NO MISTRAL_API_KEY" but .env exists

npm workspaces run the server with `cwd = server/`, so a bare `dotenv/config` looks for
`server/.env` and never sees `document-ai/.env`. Already fixed (`index.ts` loads the root one
explicitly) — but if you move files around, this is the first thing to re-check.

### A shell command can't find the PDF that obviously exists

The chemistry book's filename contains a **non-breaking space** (`\xa0`), not a regular space.
`ls`/`cp` with a pasted name fail; the browser handles it fine.

```bash
python3 -c "import glob; print(repr(glob.glob('*1-100*')[0]))"
```

Rename the file if you're going to script against it.

### The PDF pane shimmers / "vibrates"

Two feedback loops, both fixed — but if it comes back, it's one of these:

1. **The canvas fighting its own scrollbar.** Measure the *pane*, never the scrolling element:
   wider canvas → scrollbar appears → less width → narrower canvas → scrollbar goes → repeat.
   `scrollbar-gutter: stable` plus measuring `paneRef` kills it.
2. **Overlapping pdf.js renders.** Every re-render must `renderTask.cancel()` the outgoing one,
   or two paint passes interleave on one bitmap.

### A feature silently does nothing (no error, just... nothing)

Check for a `ref` on a function component. **React 18 drops it.** This exact bug killed the
scroll-spy and the page-jump box: `pageRefs` stayed empty, the IntersectionObserver had nothing
to observe, and nothing errored. Use `forwardRef`.

### The library is empty, but the stats say documents were processed (fixed July 2026)

The home screen reporting "2 pages hand-corrected · 1 doc" **directly above an empty document
grid** was not two bugs, and nothing was lost. The tiles read the *server* (`/api/edits`,
`/api/cache`); the grid read *that browser's* IndexedDB. Both were telling the truth about
different machines.

The library lived in IndexedDB, which is scoped to an **origin** — and this UI is served from
three: `:5174` standalone, `:5173` inside Relire, `:8787` built. Process a book on one, open
another, and the library reads empty while every expensive thing sits on disk, keyed and intact.
Unreachable looks exactly like lost, and costs more, because you go and pay again.

The library is now server-side SQLite (`.library/library.db`, `server/src/library.ts`), keyed by
the same content hash as the cache and the edit store. If a library still reads empty:

1. **Is the Node server up?** `curl localhost:8787/api/library` → `{"entries":[...]}`. The library
   *is* the server now; with :8787 down the grid is empty and the dropzone still works.
2. **Inside Relire, is it hitting the right server?** Relire's `/api` belongs to FastAPI (:8000);
   the playground is proxied at `/docai`. `sync-docai.mjs` rewrites `/api/` → `/docai/` for the
   files it is told about — **a new `lib/` file that fetches the Node server must be added to that
   list**, or it 404s against FastAPI and the tab reads empty. `store.ts` is on the list. Check the
   generated copy: `grep -rn "fetch(\`/api/" frontend/src/docai/lib/` must return nothing.
3. **Filed but no bytes?** `hasBytes: false` in `/api/library` means the entry's PUT landed and its
   blob PUT did not. `readEntry` returns null for those on purpose — a document with no bytes can be
   neither rendered nor re-run, and a broken row is worse than an honest absence. Re-drop the file.

Re-dropping is always safe and always cheap: cache, edits and library agree on the content hash, so
the same scan comes back with its result *and* its corrections, and never calls the API.

## Editing the code

`document-ai/web/src` is the **source of truth**. `frontend/src/docai/` is a **generated copy**
(marked `GENERATED — do not edit`) produced by `frontend/scripts/sync-docai.mjs`, which runs
automatically on `npm run dev` / `npm run build` in `frontend/`.

Edit the source. If you edit the copy, your work is deleted on the next build.

The sync script rewrites three things, and each was a bug once:

| It does | Because |
|---|---|
| Scopes all CSS under `.docai-root` | otherwise `body`, `input`, `.btn` leak into Relire's Review page |
| Copies `@keyframes` verbatim | scoping them produces invalid CSS and silently kills animations |
| Rewrites `/api/*` → `/docai/*` **globally** | Relire's `/api` is FastAPI; a non-global replace left the chunked path's *second* call 404ing |

## The thing to keep remembering

Mistral silently repairs text. `test-chimie.pdf` is printed with no accents; page 1 came back
with `composée`, `hydrogène`, `oxygène` **invented** — while `matiere` in the same heading, and
all of page 2, were left alone. It corrects some words on some pages and flags none of it.

So the playground's markdown is a **claim about** the scan, not evidence of it. That is why the
Document AI tab was added *beside* Review rather than replacing it, and why it has no `verified`
state. Don't give it one.
